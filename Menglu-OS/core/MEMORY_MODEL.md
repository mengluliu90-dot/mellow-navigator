# Memory Model

## Purpose

Define what the system should remember and how it should use context without storing sensitive evidence in this public repository.

## Memory buckets

### 1. Communication preferences

- written communication preferred
- avoid phone contact where possible
- allow extra processing time
- use clear and structured wording
- one issue at a time where possible

### 2. Accessibility needs

- reasonable adjustments
- support person involvement
- simple written summaries
- reduced time pressure
- predictable steps

### 3. Case status

Use non-sensitive summaries only:

- case title
- organisation
- status
- next step
- deadline
- risk level

### 4. Evidence map

Store categories and references only.

Do not store private evidence itself in this public repository.

### 5. Templates

Reusable wording for emails, requests, complaints, and follow-ups.

## Memory use rules

- Use memory to reduce repetition.
- Check whether new information links to existing issues.
- Avoid asking Menglu for information already available in the current working context.
- Do not overstate memory as evidence.
- If uncertain, label as Needs clarification.
- Avoid saving duplicates.
- Save stable long-term functional information only when it is likely to improve future support.
- Do not save temporary wording, one-off drafts, live case details, or information likely to change.

## Automatic memory triage

When Menglu provides substantial information, classify it before deciding whether to remember it.

### Remember

Save stable, reusable information such as:

- enduring communication needs
- long-term accessibility requirements
- functional limitations
- support needs
- stable preferences
- recurring risks
- reusable advocacy principles

### Do not remember

Do not save:

- one-off statements
- temporary incidents
- live legal or administrative details
- exact balances, dates, or identifiers unless specifically needed and safe
- private evidence documents
- information already remembered

### Ask before saving

Ask a clarification question if the information involves:

- another person's legal, medical, or financial matter
- sensitive third-party details
- unclear intent about whether it should be retained
- potentially disputed or unverified claims

## Evidence classification

Use three levels when organising evidence.

### Level 1: Permanent facts

Stable profile information and enduring needs.

Examples:

- communication preferences
- accessibility requirements
- long-term functional limitations
- confirmed diagnoses or assessments when already documented elsewhere

### Level 2: Long-term functional evidence

Real-world examples that demonstrate functional impact over time.

Examples:

- recurring service-access barriers
- documented communication breakdowns
- real financial or administrative incidents
- assistive equipment or support-session evidence

### Level 3: Case-specific evidence

Evidence used for a particular matter only.

Examples:

- a draft statement for one organisation
- a live complaint timeline
- one-off incident details
- private correspondence

Keep Level 3 evidence outside this public repository unless it is converted into a non-sensitive template or index reference.

## Travel evidence reuse rule

When travel is being planned, reuse existing stable information before asking for it again.

Relevant reusable items may include:

- communication preferences
- accessibility requirements
- support-person roles and limits
- medication and allergy information, if available in the private working context
- emergency contact structure, if available in the private working context
- medical or communication passports, if available in the private working context
- previous travel barriers or recovery patterns, if already documented

Do not recreate duplicate travel profiles unless the destination, trip type, or risk profile makes a new document necessary.

For public repository documentation, store only the reuse rule and document categories. Do not store private medical details, identifiers, contact details, travel dates, booking references, or live trip information.

## Real-world evidence rule

When preparing advocacy materials, prefer real-world examples over hypothetical statements where available.

A real-world example should be used to show:

- what happened
- what support was missing or required
- what risk occurred
- what consequence followed
- what adjustment or support would reduce future risk

Do not include sensitive details unless Menglu explicitly asks and the output destination is private and appropriate.

## Public repository boundary

This repository should contain operating instructions and non-sensitive templates only.

Private records must remain outside this repository.
