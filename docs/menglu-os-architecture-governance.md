# Menglu OS Architecture Governance

## Purpose

This document prevents Menglu OS from splitting into duplicate prompts, duplicate chats, duplicate automations, or parallel operating systems.

Treat new ideas as extensions of Menglu OS unless there is a clear reason to keep them separate.

The default goal is one coherent Menglu OS with the fewest possible prompts, workflows, automations, and duplicated documents.

## Status

Permanent Menglu OS governance layer.

Primary component: Decision Engine.

Linked components:

- Executive Function Engine
- Verification Engine
- Memory & Evidence Engine
- Communication Engine
- Output Engine
- Existing automations
- GitHub documentation

## Core Rule

Before creating any new framework, workflow, prompt, automation, document, or operating system:

1. Review existing Menglu OS material where available.
2. Identify what is genuinely new.
3. Check whether the function is already covered.
4. Integrate into the existing architecture where possible.
5. Avoid creating duplicate systems.

## Integration Review

For every significant idea, classify it under these headings.

### 1. What is new

State only the genuinely new capability.

Do not restate existing Menglu OS rules unless they are being changed.

### 2. Where it belongs

Assign the idea to one or more existing components, for example:

- Executive Function Engine
- Communication Engine
- Memory & Evidence Engine
- Decision Engine
- Verification Engine
- Output Engine
- Existing automations
- Existing GitHub documentation

### 3. Why

Explain why the idea belongs in that component instead of becoming another independent prompt, workflow, or chat.

### 4. Conflicts

Identify:

- duplication
- overlap
- contradiction
- unnecessary complexity
- stale or superseded wording
- automation overlap
- documentation overlap

### 5. Recommendation

Use one of these outcomes:

- Integrate into Menglu OS
- Replace an older workflow
- Merge with an existing workflow
- Keep as a separate module, only if there is a clear technical reason

## Verification Requirements

Before giving conclusions:

- Separate confirmed facts, unconfirmed information, assumptions, and missing information.
- Do not invent facts.
- Clearly state uncertainty when evidence is insufficient.
- Ask only the minimum number of clarification questions required.
- Prefer official or primary sources when factual verification is needed.
- Reuse existing verified information instead of recreating it.

## Automation Governance

When a repeated task is identified:

1. Check whether an existing automation already performs the same function.
2. Prefer improving an existing automation instead of creating another one.
3. Recommend consolidation where possible.
4. Avoid duplicate reminders, duplicate monitoring, and duplicate reports.
5. Design automations to work quietly and notify only when action is required.

## GitHub Documentation Governance

If a conversation results in a permanent Menglu OS improvement:

1. Recommend recording it in the Menglu OS GitHub repository.
2. Prefer updating existing documentation instead of creating duplicate documents.
3. Record only stable, reusable improvements.
4. Avoid storing temporary case details as permanent architecture unless they define a reusable rule.

## Architecture Change Log Rule

When Menglu OS changes, record the change in a single traceable location where practical.

Recommended fields:

| Field | Purpose |
| --- | --- |
| Date | When the change was made |
| Component | Which Menglu OS component changed |
| Change | What was added, removed, or revised |
| Reason | Why the change was made |
| Replaced | What older workflow, wording, or rule it superseded |
| GitHub Updated | Whether the repository was updated |

## Final Check

Before finishing every substantial Menglu OS task, check:

1. Is this already covered elsewhere?
2. Can this be merged into Menglu OS?
3. Can an existing automation be improved instead of creating a new one?
4. Can cognitive load be reduced?
5. Is every important claim supported, verifiable, clearly labelled, or explicitly marked as uncertain?

## Operating Principle

Information alone is not success.

Menglu OS succeeds when it reduces burden, reduces risk, improves accessibility, improves coordination, preserves evidence, and protects future options while preserving Menglu's autonomy and decision-making authority.
